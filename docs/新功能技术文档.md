## 功能一：错误频率告警模块

### 1. 功能概述

当系统检测到某类错误在短时间内频繁发生时，自动通过飞书机器人发送告警通知，帮助开发团队快速响应线上问题。

**核心特性：**
- ✅ 定时扫描高频错误（每 5 分钟）
- ✅ 可配置的告警阈值和时间窗口
- ✅ 控频机制，避免同一错误重复告警
- ✅ 插件化告警渠道设计，易于扩展

### 2. 架构设计


### 4. 控频机制详解

**问题：** 同一个错误可能持续发生，如果每次都发送告警，会造成告警轰炸。

**解决方案：** 使用 Redis `SET NX EX` 命令实现原子性的冷却期检查。

```typescript
// Redis Key 格式
const key = `alert:sent:${appId}:${fingerprint}`;

// 原子操作：设置 key + 过期时间 + 仅当 key 不存在时才设置
const result = await redis.set(key, timestamp, 'EX', cooldownSeconds, 'NX');

// 返回 'OK' → key 不存在，可以发送告警
// 返回 null → key 已存在，仍在冷却期内
```

**为什么用 Redis 而不是内存变量？**
1. 多实例部署：多个 Gateway 实例共享同一个 Redis
2. 持久化：重启后告警记录不丢失
3. 自动过期：Redis TTL 自动清理过期 key

### 6. 配置说明

```bash
# .env 文件配置

# 是否启用告警功能
ALERT_ENABLED=true

# 错误次数阈值：超过此数值触发告警
ALERT_THRESHOLD=100

# 统计时间窗口（分钟）
ALERT_WINDOW_MINUTES=5

# 冷却时间（分钟）：同一错误在冷却期内不重复告警
ALERT_COOLDOWN_MINUTES=30

# 飞书机器人 Webhook URL
ALERT_FEISHU_WEBHOOK=https://open.feishu.cn/...

# 飞书机器人签名密钥（可选）
ALERT_FEISHU_SECRET=your-secret
```

## 功能二：SourceMap + Git 关联

### 1. 功能概述

在错误详情中展示错误代码对应的 Git 提交信息，帮助开发者快速定位问题来源。

**核心特性：**
- ✅ CI 上传 SourceMap 时携带 Git 信息
- ✅ 错误详情页展示 Git 提交信息
- ✅ 告警消息中包含 Git 提交信息

### 2. 数据库变更

```sql
-- sourcemaps 表新增字段
ALTER TABLE sourcemaps ADD COLUMN git_commit VARCHAR(64);
ALTER TABLE sourcemaps ADD COLUMN git_author VARCHAR(128);
ALTER TABLE sourcemaps ADD COLUMN git_message TEXT;
ALTER TABLE sourcemaps ADD COLUMN git_branch VARCHAR(128);
```

### 3. CI 集成示例

```bash
#!/bin/bash
# CI 脚本：上传 SourceMap 并携带 Git 信息

# 获取 Git 信息
GIT_COMMIT=$(git rev-parse HEAD)
GIT_AUTHOR=$(git log -1 --pretty=format:'%an')
GIT_MESSAGE=$(git log -1 --pretty=format:'%s')
GIT_BRANCH=$(git rev-parse --abbrev-ref HEAD)

# 上传 SourceMap
curl -X POST http://gateway:3000/v1/sourcemap \
  -H "Content-Type: application/json" \
  -H "x-api-key: your-api-key" \
  -d "{
    \"appId\": \"my-web-app\",
    \"version\": \"$GIT_COMMIT\",
    \"filename\": \"main.js\",
    \"mapContent\": \"$(cat dist/main.js.map)\",
    \"gitCommit\": \"$GIT_COMMIT\",
    \"gitAuthor\": \"$GIT_AUTHOR\",
    \"gitMessage\": \"$GIT_MESSAGE\",
    \"gitBranch\": \"$GIT_BRANCH\"
  }"
```

## 面试要点总结

### 1. 告警模块面试要点

| 问题 | 答案要点 |
|------|----------|
| **为什么用 Redis 做控频？** | 1. 多实例共享 2. 原子操作保证并发安全 3. 自动过期无需手动清理 |
| **SET NX EX 是什么？** | SET key value EX seconds NX，原子操作：设置 key + 过期时间 + 仅当 key 不存在时才设置 |
| **为什么选择 @nestjs/schedule 而不是 Bull Queue？** | 告警任务不需要重试、不需要持久化、实现简单 |
| **多实例部署如何避免重复告警？** | Redis 控频机制：即使多实例同时扫描，也只会发送一次告警 |
| **如何扩展新的告警渠道？** | 实现 AlertChannel 接口，在 AlertModule 中注册 Provider |

### 2. Git 关联面试要点

| 问题 | 答案要点 |
|------|----------|
| **如何关联 Git 提交？** | CI 上传 SourceMap 时携带 git commit 信息，存入 sourcemaps 表 |
| **如何根据错误找到 Git 提交？** | 根据错误文件名查询 sourcemaps 表，获取最新的 Git 信息 |
| **为什么不调用 Git API？** | 简化实现，避免增加外部依赖；Git 信息在上传时已记录 |

### 3. 架构设计面试要点

| 问题 | 答案要点 |
|------|----------|
| **模块化设计的好处？** | 高内聚低耦合、便于测试、易于扩展 |
| **依赖注入的好处？** | 解耦、便于单元测试（可以 mock） |
| **策略模式的应用？** | AlertChannel 接口定义规范，不同渠道是不同策略实现 |
| **为什么用 TimescaleDB？** | 时序数据自动分区、查询性能优化、自动数据保留策略 |

### 4. 关键代码片段

**Redis 控频：**
```typescript
const result = await redis.set(key, timestamp, 'EX', cooldownSeconds, 'NX');
// 返回 'OK' → 可以发送告警
// 返回 null → 在冷却期内
```

**高频错误查询：**
```sql
SELECT fingerprint, COUNT(*) as count
FROM events
WHERE type = 'error' AND ts >= $start AND ts <= $end
GROUP BY fingerprint
HAVING COUNT(*) >= $threshold
```

**策略模式接口：**
```typescript
interface AlertChannel {
  send(payload: AlertPayload): Promise<AlertResult>;
  getType(): AlertChannelType;
  isAvailable(): boolean;
}
```
