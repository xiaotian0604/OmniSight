## 一、为什么要做这个项目

我之前在业务开发中遇到一个反复出现的问题：线上用户反馈 "页面白了" "点了没反应"，但我们排查的时候既没有报错日志，也无法复现。靠用户截图和口头描述定位 Bug，效率极低，有些偶发问题甚至拖了好几个版本都没解决。

市面上有 Sentry、Fundebug 这类成熟的监控平台，但我想**真正理解前端监控的整条链路是怎么跑通的**——从浏览器里的一个 JS 报错，到最终在控制台上看到可视化图表和用户回放录像，中间经过了哪些环节、做了哪些取舍。所以我决定从零搭一套完整的前端可观测性系统，取名叫 OmniSight（Omni 是"全方位"的意思）。

这个项目我的定位很明确：**前端深度优先，后端够用即可**。SDK 和控制台是核心，每一行代码我都要能讲清楚设计意图；后端就是 NestJS 薄薄一层，能接数据、能查询、不崩就行。不堆技术概念，每个技术选型都有明确的理由。

## 二、整体架构设计

### 2.1 技术选型与理由

整个项目用 **Turborepo + pnpm workspace** 做 Monorepo 管理，包含三个应用和两个共享包。

**为什么选 Monorepo？** 因为 SDK 的类型定义要同时被 Gateway 和 Console 引用。如果拆成多仓库，改一个类型字段要发三个 PR、等三次 CI，效率太低。Monorepo 下改一处 `shared-types`，三个应用立刻同步，类型安全在编译期就能保证。

技术栈上：

| 层级 | 技术 | 为什么选它 |
|------|------|-----------|
| SDK | TypeScript + Rollup | 手写采集逻辑，Rollup 打包体积小，支持 ESM/CJS/IIFE 三格式输出 |
| Gateway | NestJS + Bull Queue | NestJS 的模块化和依赖注入天然适合拆分上报、查询、鉴权等职责；Bull 做异步队列解耦 |
| 数据库 | PostgreSQL + TimescaleDB | 监控数据本质是时序数据，TimescaleDB 的 `time_bucket` 聚合比原生 PG 快 10 倍以上，且不需要额外部署 ClickHouse |
| 缓存/队列 | Redis | 一个 Redis 同时承担 Bull 消息队列、布隆过滤器去重、API Key 缓存三个角色，零额外部署 |
| Console | React 18 + ECharts + rrweb-player | React Query 管理服务端状态，Zustand 管理全局状态（轻量且无 Provider），ECharts 做时序图表 |

**为什么不用 Kafka？** 数据量没到那个级别，Bull + Redis 已经够用，而且零额外部署——面试现场 `docker-compose up -d` 就能把 PG 和 Redis 全起来。

**为什么不用 ClickHouse？** 同样的道理，TimescaleDB 是 PG 的插件，用标准 SQL 就能查，不需要额外学一套 ClickHouse 的语法和运维体系。对于这个项目的数据量，性能绰绰有余。

### 2.2 数据流转链路

整条数据流是这样跑的：

```
浏览器 → SDK 采集事件 → 采样/去重 → 批量上报（Beacon API / XHR）
    → Gateway 接收 → 鉴权（API Key）→ 写入 Bull Queue → 立即返回 200
    → Worker 异步消费 → Redis 布隆过滤器二次去重 → 写入 PostgreSQL
    → Console 拉取查询 → React Query 缓存 → ECharts 渲染图表
```

**为什么上报和入库要解耦？** 因为数据库写入是慢操作，如果 SDK 上报一条数据，Gateway 同步写 PG 再返回，接口延迟会很高。用 Bull Queue 做异步，Gateway 只负责把事件扔进队列就返回 200，Worker 在后台慢慢消费写库。这样上报接口的 P99 能压到 50ms 以内，SDK 端几乎感知不到延迟。

## 三、SDK 设计（核心模块，最花心思的部分）

SDK 是我花时间最多的部分，最终打包体积控制在 15KB gzip 以内。设计上分四层：**采集层 → 采样/去重层 → 上报层 → 隐私层**。

### 3.1 采集层（7 个采集器）

每个采集器职责单一，初始化时返回 cleanup 函数，SDK 销毁时统一调用：

**错误采集（error.ts）**：挂了三个钩子——`window.onerror` 捕获同步运行时错误，`unhandledrejection` 捕获未处理的 Promise 异常，劫持 `console.error` 捕获业务主动抛出的错误。这里有个细节：`window.onerror` 也会收到资源加载错误（比如图片 404），我通过检查 `event.error` 是否为 null 来区分——JS 错误的 `event.error` 是 Error 对象，资源加载错误是 null。

**接口采集（api.ts）**：劫持了 XHR 和 Fetch 两种请求方式。XHR 是重写 `open` 和 `send` 方法，在 `loadend` 事件里计算耗时；Fetch 是包装 `window.fetch`，在 then/catch 里计算耗时。有一个关键问题要处理：**SDK 自身的上报请求不能被采集**，否则会无限循环。我写了一个 `isSdkRequest()` 函数，检查 URL 是否包含 dsn 配置的地址且路径包含 `/v1/`，是的话就跳过。另外我用 `performance.now()` 而不是 `Date.now()` 来计算耗时，精度从毫秒级提升到了微秒级。

**Web Vitals 采集（vitals.ts）**：采集 LCP、CLS、TTFB、INP 四个核心指标。这里用了 `import('web-vitals')` 动态导入，这样如果业务方没安装 web-vitals 这个包，SDK 不会崩溃，只是静默跳过。这种可选依赖的设计对 SDK 的通用性很重要。

**资源采集（resource.ts）**：用 `PerformanceObserver` 监听 `resource` 类型的性能条目，采集脚本、样式、图片、字体等静态资源的加载耗时。比轮询 `performance.getEntriesByType` 高效得多，而且设置了 `buffered: true`，能回溯 Observer 注册前就已经加载完的资源。

**行为采集（behavior.ts）**：采集点击和路由变化，作为"面包屑"数据。点击事件我监听的是捕获阶段（`addEventListener` 第三个参数传 `true`），这样即使业务代码调用了 `stopPropagation()`，我也能收到。路由变化比较麻烦，`popstate` 只管前进后退，`pushState` 和 `replaceState` 是不触发任何事件的，所以我劫持了 `history.pushState` 和 `history.replaceState` 的原型方法。

**白屏检测（whitescreen.ts）**：页面加载 3 秒后检查 `#root`（React）或 `#app`（Vue）容器有没有子元素，没有就判定白屏。用 `children.length` 而不是 `childNodes.length`，因为后者会把空白文本节点也算进去，产生误判。

**录制采集（replay.ts）**：基于 rrweb 录制用户操作。这里的策略是**错误窗口模式**——rrweb 持续录制，但只保留最近 30 秒的事件在内存里（Ring Buffer），平时不上传。当错误发生后，再继续录制 10 秒，然后把这 40 秒的录像上传。这样存储成本降低了大约 80%，因为绝大部分正常操作的录像都不需要上传。

### 3.2 采样层

不是所有事件都需要上报。我设计了一个**双轨采样策略**：

- **关键事件 100% 采集**：JS 错误、白屏、接口耗时 > 3 秒，这三类不受采样率影响，一条都不能丢
- **普通事件按比例采样**：默认 10%，基于 `sessionId` 的哈希值采样

为什么用 sessionId 哈希而不是 `Math.random()`？因为随机采样会导致同一用户的行为数据碎片化——点击事件被采集了，但紧接着的路由变化被跳过了，数据链路断了。基于 sessionId 哈希，同一用户在同一会话中的所有事件，要么全部采集，要么全部跳过，保证行为数据的完整性。

哈希算法用的是 djb2 变体，`(hash << 5) - hash + charCode`，等价于 `hash * 31 + charCode`，位运算比乘法更快。把哈希结果映射到 [0, 1) 区间后和采样率比较。

### 3.3 去重层

错误事件还有一层 LRU 去重。假设一个 `for` 循环里有个 Bug，每次迭代都抛同样的错误，不去重的话一秒能上报几百条一模一样的数据。

我的做法是用 `Map` 实现了一个 LRU 缓存（利用 Map 的插入顺序特性），存储错误指纹。60 秒内相同指纹的错误只上报一次。指纹的生成规则是 `message + 堆栈第一帧` 做 djb2 哈希——同一个位置抛出的同一个错误消息，指纹一定相同。

Map 实现 LRU 的技巧：命中时先 `delete` 再 `set`，把条目移到 Map 末尾；淘汰时用 `keys().next().value` 取最旧的条目删掉。比用双向链表简洁得多。

### 3.4 上报层

事件不是产生一条就发一条，而是**批量上报**。队列满 20 条或者距上次上报超过 5 秒，触发一次 flush。优先用 `navigator.sendBeacon`，因为它在页面关闭时也能可靠发送；如果 Beacon 不可用或数据超过 64KB，降级用 XHR。

这里 XHR 降级有个设计考虑：为什么不用 fetch？因为 SDK 已经劫持了 `window.fetch` 来采集接口数据。如果上报也用 fetch，就会触发自己的 API 采集器，虽然 `isSdkRequest()` 能过滤掉，但还是增加了不必要的开销。直接用原生 XHR 从根源上避免了这个问题。

`beforeunload` 事件时会强制 flush 一次剩余队列，确保用户关闭页面前的数据不丢失。

### 3.5 隐私层

如果用户传入了 userId，我会用 SHA-256 做单向哈希后存储，原始 ID 永远不会离开客户端。优先使用浏览器的 Web Crypto API（`crypto.subtle.digest`），不支持时降级为双重 djb2 哈希。rrweb 录制时也做了脱敏处理——密码框始终遮盖，`[data-no-record]` 标记的元素不录制。

## 四、Gateway 设计

Gateway 用 NestJS 搭建，模块化设计，按职责拆分为 6 个模块。

### 4.1 上报模块（ingest/）

`POST /v1/ingest/batch` 接收 SDK 的批量上报数据。请求体是一个 JSON 数组，用 `class-validator` 做 DTO 校验（type、appId、sessionId、ts 是必填字段），校验通过后直接写入 Bull Queue，立即返回 `{ success: true, jobId }`。

这里有个设计决定：`ValidationPipe` 的 `whitelist` 设为 `false`。因为不同类型的事件有不同的字段——error 事件有 `message` 和 `stack`，api 事件有 `apiUrl` 和 `duration`，vital 事件有 `name` 和 `value`。如果开启了 whitelist，这些 DTO 里没声明的字段会被自动剥离掉，数据就丢了。

### 4.2 队列消费（worker/）

Worker 从 Bull Queue 中取出事件，做两件事：

1. **Redis 去重**：用 `SET key NX EX 3600` 模拟布隆过滤器。key 是 `dedup:{appId}:{fingerprint}`，如果 SET 成功说明是新事件，SET 失败说明 1 小时内已经入库过了。只对有 fingerprint 的事件（主要是 error）做去重。
2. **写入 PostgreSQL**：`INSERT INTO events`，字段包括 app_id、session_id、type、ts、fingerprint、payload（JSONB 存完整事件数据）。

Bull 配置了失败重试 3 次，指数退避（2s、4s、8s），成功后自动删除 Job。

### 4.3 查询模块（query/）

提供给 Console 的数据查询接口：

- **错误聚合列表**：`GROUP BY fingerprint`，计算每个错误的出现次数和影响用户数（`COUNT(DISTINCT session_id)`）
- **错误率时序**：用 TimescaleDB 的 `time_bucket` 做时间聚合，统计每个时间桶内的错误数和总事件数。如果 TimescaleDB 不可用，自动降级到 PG 原生的 `date_trunc`
- **API 指标**：`percentile_cont(0.50/0.75/0.99)` 计算接口耗时的 P50、P75、P99，还有错误率
- **Web Vitals 时序**：按指标名分组，计算每个时间桶的平均值

### 4.4 鉴权模块（auth/）

所有上报接口都需要 API Key 鉴权。实现是一个 NestJS Guard，检查请求头的 `x-api-key`：

1. 先查 Redis 缓存（`apikey:{key}`），命中直接通过
2. 未命中查 `projects` 表
3. 查到了写入 Redis（TTL 5 分钟）再通过
4. 没查到返回 401

两层缓存显著减少了数据库压力，大部分请求在 Redis 层就能完成校验。

### 4.5 录像模块（replay/）

`POST /v1/replay` 接收 rrweb 录像数据。用 `ON CONFLICT (session_id) DO UPDATE` 做 upsert——同一个 session 可能分多次上传录像片段，用 JSONB 的 `||` 操作符合并事件数组，而不是覆盖。

### 4.6 SourceMap 模块（sourcemap/）

`POST /v1/sourcemap` 接收 CI 上传的 SourceMap 文件。这里做了**路径穿越防护**：`appId`、`version`、`filename` 三个字段都用正则 `^[a-zA-Z0-9._-]+$` 校验，禁止包含 `..`、`/`、`\` 等危险字符，防止攻击者通过构造路径覆盖服务器上的任意文件。

## 五、Console 设计

Console 是 React 18 单页应用，用 Vite 做开发服务器和构建。

### 5.1 状态管理策略

我把状态分成了三种，用不同的方案管理：

| 状态类型 | 方案 | 举例 |
|---------|------|------|
| 服务端状态 | React Query | 错误列表、指标数据、回放录像 |
| 全局客户端状态 | Zustand | 当前 appId、时间范围、告警列表 |
| 局部状态 | useState | 排序方式、播放状态、表单输入 |

为什么选 Zustand 而不是 Redux？因为 Zustand 不需要 Provider 包裹，store 可以在 React 组件外直接调用 `getState()`——比如我在 WebSocket 的 `onmessage` 回调里就需要往 alert store 里添加告警，如果用 Redux 还得用 `store.dispatch`，Zustand 直接 `useAlertStore.getState().addAlert()` 就行。

### 5.2 核心页面

**概览页**：三个组件——错误率折线图（时间粒度自动适配，1 小时用 5 分钟聚合，7 天用 1 天聚合）、Web Vitals 评分卡（颜色按 Google 标准，good 绿色、needs-improvement 黄色、poor 红色）、Top 10 错误表。

**错误详情页**：展示错误堆栈、用户行为面包屑（点击 → 路由跳转 → XHR 请求 → 报错，完整链路）、关联回放链接。堆栈解析支持 Chrome 和 Firefox 两种格式。

**回放页**：用 rrweb-player 做视频式回放，支持播放/暂停、倍速、时间轴跳转。右侧是事件时间线（从 rrweb Custom Event 中提取），点击事件标记可以跳转到对应时刻。

**性能页**：Web Vitals 趋势图（CLS 用右侧 Y 轴，因为量级比 LCP/TTFB 小得多）、API 接口耗时表（P99 > 3s 标红）。

### 5.3 实时告警

通过 WebSocket 接收 Gateway 推送的告警。连接断开后**指数退避重连**（1s、2s、4s...最大 30s），每 30 秒发一次心跳保活。`addAlert` 函数用 `useRef` 包裹后传入 `connect` 回调，避免因为 Zustand 引用变化导致 `useEffect` 反复重连。

## 六、亮点与难点

### 亮点

1. **完整的数据链路**：从浏览器采集到数据库存储到可视化展示，全链路打通，不是 Demo 级别的拼接
2. **SDK 体积 < 15KB gzip**：手写采集逻辑，rrweb 和 web-vitals 做可选依赖动态导入，不打进 bundle
3. **双层去重**：客户端 LRU + 服务端 Redis，避免重复数据浪费存储和带宽
4. **错误窗口录像策略**：平时只录不传，错误时才上传前后 40 秒，存储成本降低约 80%
5. **sessionId 哈希采样**：保证同一用户行为数据的完整性，避免随机采样的碎片化
6. **上报与入库解耦**：Bull Queue 异步消费，上报 P99 < 50ms

### 难点

1. **XHR/Fetch 劫持的自循环问题**：SDK 劫持了 Fetch 采集接口数据，但 SDK 自身的上报也需要发请求。解决方案是 `isSdkRequest()` 过滤 + 上报降级用原生 XHR 而非 Fetch。

2. **rrweb 录制的性能与存储权衡**：全量录制上传数据量巨大。Ring Buffer 只保留 30 秒 + 错误触发才上传，在故障复现能力和资源消耗之间找到了平衡。

3. **TimescaleDB 兼容性降级**：不是所有环境都安装了 TimescaleDB 扩展。查询层先尝试 `time_bucket`，catch 到特定错误后自动降级到 `date_trunc`，还根据用户请求的时间间隔映射到合适的 truncation 精度。

4. **多类型事件的 DTO 校验**：SDK 上报的事件有 6 种类型，每种字段不同。如果用 `whitelist: true`，动态字段会被剥离；如果完全不校验，又有安全风险。最终用 `whitelist: false` + 核心字段强校验的方案。

5. **rrweb-player 的 API 适配**：rrweb-player v2 alpha 的 API 和文档不一致，`play()`、`pause()`、`setSpeed()` 等方法需要通过 `getReplayer()` 拿到底层 Replayer 实例才能调用，`getCurrentTime()` 也是。花了不少时间翻源码才搞定。
